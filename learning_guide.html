Iteration 1 — The Big Picture

What Did We Do and Why Does It Matter?

The Problem

Hate speech on social media disproportionately affects non-English communities. Automated detection systems are almost entirely built for English. Urdu, spoken by over 230 million people, is the national language of Pakistan and widely used on Twitter — but almost entirely unserved by existing hate speech detection tools.

What is ISE-Hate? A corpus of 21,759 Pakistani tweets in Nastaliq (Arabic) script, collected between March–August 2022 on hashtags related to interfaith, sectarian, and ethnic conflict. Annotated by five bilingual Pakistani native speakers. 39.8% hate, 60.2% neutral. Provides both binary labels and sub-category labels (Interfaith, Sectarian, Ethnic, Other).

Our Approach

We used Transfer Learning: fine-tune AI models on English hate speech data, then evaluate how well they detect Urdu hate speech — with zero, few, or full Urdu supervision.

Five Key Findings

Finding 1 — Larger model = better zero-shot Urdu.
XLM-R-large is the best at zero-shot Urdu (hate F1 = 0.147), not the worst. More capacity means better Urdu representations in the model.
Finding 2 — Zero-shot fails due to class-imbalance bias, not language gap.
The failure mode is low recall (0.080), not low precision (0.885). The model knows hate when it sees English-style hate — it just almost never fires on Urdu text.
Finding 3 — Few-shot works stably on Nastaliq Urdu.
Hate F1 rises monotonically from 0.590 (K=8) to 0.694 (K=64). No collapse. Correct class weights are critical.
Finding 4 — 20× zero-shot fairness disparity.
Interfaith hate recall = 28.2% vs "Other" hate recall = 1.4%. English annotator bias propagates into systematic Urdu blind spots.
Finding 5 — Supervised XLM-R-large gives modest gains at high FPR cost.
Hate F1 = 0.796 vs TF-IDF+LR's 0.747. But false positive rate rises from 0.154 to 0.228. Dominant signal in ISE-Hate is lexical — deep representations add limited marginal value.

6. Why Hate F1, Not Accuracy?

On ISE-Hate (60% non-hate), a model that never predicts hate gets 60% accuracy but hate F1 = 0. Hate F1 = 2 × Precision × Recall / (Precision + Recall). It requires the model to be both precise (not over-flagging) and recall-effective (not missing hate). A model that predicts everything as non-hate gets F1_hate = 0. Macro F1 averages F1 across both classes — penalising failure on either.
Iteration 3 — Under the Hood

Transformers, Training, and the Failure Mode

1. Why the Zero-Shot Failure is Low Recall, Not Low Precision

XLM-R-large zero-shot: Precision = 0.885, Recall = 0.080.
The model can identify Urdu hate when it fires — precision is near-perfect. But it almost never fires. This is the signature of a miscalibrated decision threshold, not an inability to understand Urdu. The model learned "predict hate only when signals are very strong (English-style hate signals)" — and Urdu hate speech has no such signals.

2. How Tokenisation Works for Nastaliq Urdu

XLM-R uses SentencePiece BPE — a vocabulary of 250,000 subword tokens trained on CommonCrawl web text from 100 languages. Nastaliq Urdu tokens are included because CommonCrawl crawled Pakistani websites. An Urdu word like نفرت (hate) is split into subword pieces that are actual known tokens in the vocabulary — unlike out-of-vocabulary Latin-script languages, Nastaliq Urdu is represented at the subword level. This is why zero-shot partially works (0.147 hate F1) rather than completely failing (near 0).

3. Class-Weighted Loss — Solving the Imbalance

Standard cross-entropy (English data, 5.2% hate): Model says "non-hate" always → loss ≈ 0.05 → never learns hate Weighted cross-entropy (w = [1.0, 8.0]): L = −Σ w[yᵢ] · log p(yᵢ | xᵢ) Missing a hate sample costs 8× more → model MUST learn hate features to reduce total loss For few-shot Urdu (39.8% hate), we use w = [1.0, 2.5]: Calibrated for the target distribution. Using [1.0, 8.0] on 40% hate data would massively over-weight hate → model would over-predict hate → high FPR.

4. Why FuzzyWuzzy Has Zero Effect on ISE-Hate

FuzzyWuzzy computes fuzzy string similarity between tweet text and a lexicon of hate phrases. The lexicon is built for Latin-script text. ISE-Hate is written in Nastaliq Arabic script (right-to-left). String matching between Arabic-script Urdu and Latin-script hate phrases produces near-zero similarity for every tweet. The added features are indistinguishable from noise. This directly contrasts with Roman Urdu, where FuzzyWuzzy was shown beneficial (Ullah et al. 2024) because the lexicon and target text share the same script.

5. K=16 Instability — Why It Happens

K=16 training, class weights [1.0, 2.5], Urdu 40% hate: Seed 0: 16 examples sampled with 8 hate, 8 non-hate (ideal) → Model learns discriminatively → macro F1 ≈ 0.68 Seed 1: 16 examples sampled with 10 hate, 6 non-hate (unlucky) → Combined with w_hate=2.5, hate loss dominates → Gradient pushes model toward over-predicting hate → FPR = 0.633 → macro F1 ≈ 0.41 Std across seeds = 0.135 (high variance) K=64 eliminates this: diverse enough to produce stable gradients. Std = 0.001 at K=64 — essentially deterministic.

6. The Script Mismatch Lesson (Nastaliq vs Latin)

Nastaliq Urdu: Arabic script, right-to-left, standardised — IS in XLM-R's training. Zero-shot hate F1 = 0.147. Few-shot is stable and monotonically improving.

Roman Urdu: Latin script, non-standard spelling, informal — NOT in XLM-R's training. Zero-shot would produce near-zero F1. Few-shot with randomly-initialised head would collapse without careful design.

The correct target language (Nastaliq Urdu) is what enables our positive few-shot results.

File 1: scripts/train.py — English Fine-Tuning with WeightedTrainer

# Subclass HuggingFace Trainer to inject class-weighted loss
class WeightedTrainer(Trainer):
    def compute_loss(self, model, inputs, return_outputs=False, **kwargs):
        labels = inputs.pop('labels')     # pull labels before forward pass
        outputs = model(**inputs)           # forward pass → logits
        weights = torch.tensor(
            [1.0, 8.0],                     # EN: non-hate=1.0, hate=8.0
            device=outputs.logits.device    # MUST match model device (GPU)
        )
        loss = nn.CrossEntropyLoss(weight=weights)(
            outputs.logits, labels
        )
        return (loss, outputs) if return_outputs else loss

# Test-set leakage prevention: zero_shot_eval.py is run separately after training

File 3: src/evaluator.py — Level-2 Fairness Metrics

# Join test predictions with ISE-Hate Level-2 labels via Tweet_ID
def level2_fairness(predictions, level2_labels):
    for subtype in ['I', 'S', 'E', 'O']:   # Interfaith/Sectarian/Ethnic/Other
        # Filter: only hate tweets in this sub-category
        group = [p for p in predictions
                 if p['true'] == 1 and level2_labels[p['id']] == subtype]
        # Recall = how often we correctly predict these as hate
        recall = sum(p['pred'] == 1 for p in group) / len(group)
        results[subtype] = {'n': len(group), 'recall': recall}

# FPR on neutral subset — how often non-hate is wrongly flagged
neutral = [p for p in predictions if p['true'] == 0]
fpr = sum(p['pred'] == 1 for p in neutral) / len(neutral)

2. The Class-Imbalance Prior — A Mathematical Argument

After English fine-tuning, the model has learned: p(hate | x_EN_hate) ≈ high (English hate text) p(hate | x_EN_nonhate) ≈ low (English non-hate text) At zero-shot on Urdu text x_UR: p(hate | x_UR) ≈ ??? The model must "decide" using its English-calibrated decision boundary. Urdu hate text has NO English hate surface features. The model falls back to the prior: p(hate) ≈ 0.052 (from training distribution). This is why precision = 0.885 (correctly high — the rare times it fires, it is right) but recall = 0.080 (it almost never fires — the prior dominates). The "fix" is not a better model; it is Urdu training data that shifts the model's decision boundary for Urdu inputs.

3. Learning Rate Schedule — Exact Numbers

English training (XLM-R-large): 21,159 samples · batch size 8 · grad accum 4 steps per epoch = ⌈21159 / 8⌉ = 2645 update steps / epoch = ⌈2645 / 4⌉ = 662 total updates (5 ep) = 662 × 5 = 3310 warmup steps = 0.1 × 3310 = 331 LR schedule (linear warmup + linear decay): LR(t) = lr_max × t/331 for t ≤ 331 (warmup) LR(t) = lr_max × (3310-t)/2979 for t > 331 (decay) Supervised ISE (1 epoch, 15,231 samples): steps per epoch = ⌈15231 / 8⌉ = 1904 update steps / epoch = ⌈1904 / 4⌉ = 476 warmup steps = 0.1 × 476 = 48

4. fp16 and Automatic Mixed Precision

FormatBytes/numberXLM-R-large weightsGradient risk
fp324 bytes560M × 4 = 2.24 GBNone
fp16 (we use)2 bytes560M × 2 = 1.12 GBUnderflow at small gradients
AMP loss scaling fix: Multiply loss by 1024 before backprop → gradients are 1024× larger → no underflow in fp16 range. Divide by 1024 after accumulation before weight update. This is automatic in PyTorch's GradScaler.

5. Gradient Checkpointing — Memory vs Speed

Normal training (24 layers, seq_len=128, hidden=1024): Store all activations: 24 × 128 × 1024 × 4 bytes ≈ 12.6 MB per sample Batch of 8: ~100 MB just for activations (plus model weights, gradients) Gradient checkpointing: Discard activations after each layer's forward pass Recompute during backward pass (forward runs ~1.33× total) Memory: O(1) layer activations instead of O(24) Cost: ~33% slower, saves ~30% peak VRAM Required for XLM-R-large (560M params + optimizer states + activations) to fit within 12 GB VRAM at batch size 8.

Quick Reference Glossary

Pre-trainingTraining on massive unlabelled text using self-supervised objectives (MLM). Builds language representations. Done once by Meta/Google.
Fine-tuningTaking pre-trained weights and training on labelled task data with a small learning rate. Updates all parameters.
Zero-shot transferApplying an English-trained model to Urdu without any Urdu training examples.
K-shot (few-shot)Giving the model K labelled Urdu examples to adapt its decision boundary for the target language.
Hate F12×P×R/(P+R) for the hate class. Primary metric — penalises both missing hate (low recall) and false alarms (low precision).
FPR (False Positive Rate)FP / (FP + TN). How often neutral content is wrongly flagged as hate. Fairness metric.
Class-weighted lossw_hate × loss_hate + w_nonhate × loss_nonhate. Forces model to prioritise hate class despite class imbalance.
Nastaliq scriptThe Arabic-script writing system for Urdu. Right-to-left, standardised. Present in XLM-R's 100-language pretraining corpus.
Gradient accumulationAccumulate gradients over N mini-batches before updating. Simulates batch size N×batch without extra VRAM.
fp16Half-precision training: 2 bytes/number instead of 4. Halves VRAM, 2× faster. Requires AMP loss scaling.
Gradient checkpointingDiscard activations during forward pass; recompute during backward. Saves ~30% VRAM, costs ~33% speed.
ISE-Hate Level-2Sub-categories of hate in ISE-Hate: Interfaith (I), Sectarian (S), Ethnic (E), Other (O). Used for fairness analysis.
Davidson et al. biasAMT workers (white, US-based) over-labelled AAE dialect features and religious language as hate. Propagates cross-lingually.
SentencePiece BPESubword tokenizer: builds vocabulary by iterative merging of frequent character pairs. 250K vocab covers 100 languages including Nastaliq Urdu.
